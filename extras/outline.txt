# 📘 MFE WORKFLOW DESIGN — ENTERPRISE ORDER PLATFORM

## 🎯 SYSTEM GOAL

Provide a **modular enterprise admin platform** where independent MFEs collaborate via **events**, not direct dependencies.

---

## 🧱 MFEs IN THE SYSTEM (4 TOTAL)

1️⃣ **Orders MFE** (Core domain)
2️⃣ **Inventory MFE** (Supporting domain)
3️⃣ **Dashboard MFE** (Aggregated view)
4️⃣ **Notifications MFE** (Awareness & audit)

---

# 1️⃣ ORDERS MFE (DOMAIN OWNER)

### 🎯 Responsibility

* Owns **order lifecycle**
* Source of truth for orders

### 📦 Data Owned

```ts
Order {
  id: string
  customer: string
  items: { sku: string, qty: number }[]
  status: CREATED | APPROVED | SHIPPED | CANCELLED
}
```

Stored in:

* `localStorage: orders`

---

### 👤 User Actions

| Action        | Description            |
| ------------- | ---------------------- |
| Create Order  | Create new order       |
| Approve Order | Commit order           |
| Cancel Order  | Cancel before shipping |
| View Orders   | Read-only list         |

---

### 🔔 Events Emitted

| Event             | When           |
| ----------------- | -------------- |
| `ORDER_CREATED`   | After creation |
| `ORDER_APPROVED`  | After approval |
| `ORDER_CANCELLED` | After cancel   |

```ts
emit({
  type: 'ORDER_APPROVED',
  payload: { orderId }
});
```

---

### ❌ What Orders MFE MUST NOT DO

* Manage inventory
* Calculate KPIs
* Show global notifications logic

---

# 2️⃣ INVENTORY MFE (RESOURCE OWNER)

### 🎯 Responsibility

* Track stock levels
* Reserve & release inventory

### 📦 Data Owned

```ts
InventoryItem {
  sku: string
  available: number
  reserved: number
}
```

Stored in:

* `localStorage: inventory`

---

### 👤 User Actions

| Action         | Description       |
| -------------- | ----------------- |
| View Inventory | Stock overview    |
| Restock        | Add quantity      |
| Adjust Stock   | Manual correction |

---

### 🔄 Reacts To Events

| Event             | Reaction      |
| ----------------- | ------------- |
| `ORDER_APPROVED`  | Reserve stock |
| `ORDER_CANCELLED` | Release stock |

```ts
if (event.type === 'ORDER_APPROVED') {
  reserveStock(event.payload.orderId);
}
```

---

### 🔔 Events Emitted

| Event                | When              |
| -------------------- | ----------------- |
| `INVENTORY_RESERVED` | Stock reserved    |
| `LOW_STOCK_ALERT`    | Threshold crossed |

---

### ❌ What Inventory MFE MUST NOT DO

* Create orders
* Display dashboards
* Show toasts directly

---

# 3️⃣ DASHBOARD MFE (AGGREGATOR)

### 🎯 Responsibility

* Read-only insights
* KPIs & health indicators

### 📦 Data Used (Not Owned)

* Orders summary
* Inventory summary
* System events

---

### 👤 User Actions

| Action        | Description     |
| ------------- | --------------- |
| View KPIs     | Totals & trends |
| View Activity | Recent events   |

---

### 🔄 Reacts To Events

| Event             | Effect       |
| ----------------- | ------------ |
| `ORDER_CREATED`   | Update count |
| `ORDER_APPROVED`  | KPI update   |
| `LOW_STOCK_ALERT` | Show warning |

---

### ❌ What Dashboard MFE MUST NOT DO

* Mutate data
* Trigger workflows

---

# 4️⃣ NOTIFICATIONS MFE (AWARENESS & AUDIT)

### 🎯 Responsibility

* User feedback
* System activity
* Audit trail

---

### 📦 Data Owned

```ts
Notification {
  id: string
  message: string
  type: INFO | WARNING | ERROR
  timestamp: number
}
```

Stored in:

* `localStorage: notifications`

---

### 👤 User Actions

| Action             | Description   |
| ------------------ | ------------- |
| View Notifications | Activity list |
| Mark Read          | Acknowledge   |

---

### 🔄 Reacts To Events

| Event             | Result               |
| ----------------- | -------------------- |
| `ORDER_CREATED`   | Info notification    |
| `ORDER_APPROVED`  | Success notification |
| `LOW_STOCK_ALERT` | Warning alert        |

---

### ❌ What Notifications MFE MUST NOT DO

* Business logic
* Data ownership

---

# 🔁 END-TO-END USER SCENARIOS

## 🧪 Scenario A — Happy Path

1. User creates order (Orders)
2. `ORDER_CREATED` emitted
3. Dashboard updates count
4. Notification logged

---

## 🧪 Scenario B — Approval Path

1. Order approved (Orders)
2. Inventory reserves stock
3. Dashboard KPI updates
4. Notification shows success

---

## 🧪 Scenario C — Risk Path

1. Inventory drops below threshold
2. `LOW_STOCK_ALERT` emitted
3. Orders block approvals
4. Dashboard shows warning
5. Notification alerts user

---

# 🧠 WHY THIS DESIGN IS STRONG

✔ Clear ownership
✔ Event-driven (no tight coupling)
✔ Easy to explain
✔ Scales naturally
✔ Perfect for MFEs

This is **exactly how large frontend systems are designed**.
